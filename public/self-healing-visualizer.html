Self-Healing Framework Visualizer

This page explains the iterative Task 3 approach. Version 1 is intentionally naive: a broken locator fails first, the DOM is attached to an xAI request, and the returned selector is validated before the test continues.

Step 1 Locator breaks #submit-login-button-does-not-exist
Step 2 DOM is attached

Interactive controls are compacted into a readable snapshot.

Step 3 xAI repairs

The model returns a selector, confidence, and reason.

Step 4 Test continues

The selector is validated, clicked, logged, and attached to the test result.

Prompt Payload

{
  "brokenSelector": "#submit-login-button-does-not-exist",
  "intent": "The primary submit button for the login form",
  "domSnapshot": "1. parent=form#login-form <button id=\"actual-login-submit\" data-testid=\"login-submit\"> text=\"Sign in\""
}

Expected Repair

{
  "selector": "#actual-login-submit",
  "confidence": 0.98,
  "reason": "The button is the submit control inside the login form."
}
Framework Shared service

SelfHealingService owns DOM capture, xAI repair, validation, and logging.

Fixture One test API

Tests import healing from a Playwright fixture instead of creating helpers manually.

Evidence Central attachment

Heal events are attached to Playwright output from the fixture after the test completes.

Next Guardrails later

Confidence gates, cache validation, and refusal rules are planned for the next version.
